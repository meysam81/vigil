# Content-Security-Policy-Report-Only -- MDN Reference

> Source: <https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Content-Security-Policy-Report-Only>
> Spec Status: Living Standard (CSP Level 3)
> Last Fetched: 2026-03-16

## Overview

The HTTP `Content-Security-Policy-Report-Only` response header allows web developers to monitor Content Security Policy violations and their effects without enforcing the policies. Violations are reported but resources are not blocked, making it ideal for testing and refining CSP rules before full enforcement via `Content-Security-Policy`.

## Behavior

- Resources that violate the policy are **not blocked** -- they load normally.
- Violation reports are still generated and sent to the configured reporting endpoint.
- Both `Content-Security-Policy` and `Content-Security-Policy-Report-Only` headers can be present simultaneously on the same response. The enforcing header blocks violations per its policy, while the report-only header independently reports violations per its own policy.
- This is the standard approach for safely testing new or tighter CSP policies before enforcement: deploy as report-only, monitor the reports, fix violations, then switch to enforcing.
- The CSP `report-to` (or deprecated `report-uri`) directive **must** be specified within the header for reports to actually be sent. Without a reporting directive, the header has no observable effect.

## Syntax

```
Content-Security-Policy-Report-Only: <policy-directive>; ...; <policy-directive>; report-to <endpoint-name>
```

## Report Format

Reports generated by `Content-Security-Policy-Report-Only` are identical in structure to those generated by the enforcing `Content-Security-Policy` header. The distinguishing factor is the `disposition` field:

- **Modern Reporting API format** (`report-to`): The report body includes `"disposition": "report"` (as opposed to `"enforce"` for enforcing CSP).
- **Legacy format** (`report-uri`): The `csp-report` JSON object includes `"disposition": "report"`.

The `disposition` field is present in both modern and legacy report formats and is the authoritative way to distinguish report-only violations from enforced violations.

## Usage with report-to and report-uri

`Content-Security-Policy-Report-Only` **requires** a reporting directive (`report-to` or `report-uri`) to serve any purpose. Without one, violations are silently discarded.

### Modern approach (report-to)

The `report-to` directive works with the Reporting API. An endpoint must first be defined via the `Reporting-Endpoints` HTTP response header, then referenced by name in the CSP:

```
Reporting-Endpoints: csp-endpoint="https://example.com/csp-reports"
Content-Security-Policy-Report-Only: default-src https:; report-to csp-endpoint
```

### Legacy approach (report-uri)

The deprecated `report-uri` directive sends reports as HTTP POST requests with `Content-Type: application/csp-report`:

```
Content-Security-Policy-Report-Only: default-src https:; report-uri /csp-report-url/
```

### Both together (recommended for cross-browser coverage)

Since `report-to` does not yet have full cross-browser support, MDN recommends specifying both directives. In browsers that support `report-to`, the `report-uri` directive is ignored:

```
Content-Security-Policy-Report-Only: default-src https:; report-uri /csp-report-url/; report-to csp-endpoint
```

## Examples

From MDN: a page with the following headers:

```http
Reporting-Endpoints: csp-endpoint="https://example.com/csp-reports"
Content-Security-Policy-Report-Only: default-src https:;
  report-uri /csp-report-url/;
  report-to csp-endpoint;
```

This policy reports violations when:

- The page loads resources over insecure HTTP (violating `default-src https:`).
- The page loads from inline scripts or styles (violating `default-src https:` which does not include `'unsafe-inline'`).

Resources are **not blocked** -- they continue to load normally. Violation reports are sent to the configured endpoint(s).

## Meta Tag Limitation

`Content-Security-Policy-Report-Only` **cannot** be set via a `<meta>` HTML element. It must be delivered as an HTTP response header. This is explicitly stated on the MDN page.

## Browser Compatibility

MDN marks this header as **Baseline Widely available**, established across all major browsers since August 2016. It is supported in Chrome, Firefox, Safari, and Edge.

Consult the [MDN compatibility table](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Content-Security-Policy-Report-Only#browser_compatibility) for current data.

## Relevance to Vigil

- Vigil receives reports from **both** enforcing (`Content-Security-Policy`) and report-only (`Content-Security-Policy-Report-Only`) policies. Both produce structurally identical violation reports.
- The `disposition` field in both modern and legacy reports distinguishes `"enforce"` vs `"report"`, allowing Vigil to classify the source policy type.
- Report-only deployments may generate **higher volume** of reports than enforcing CSP. Since resources are not blocked, users continue navigating and triggering additional violations that would otherwise be stopped by enforcement.
- Sites commonly use report-only as the first step in CSP adoption: deploy report-only, analyze reports in Vigil, fix violations, then switch to enforcing. This means Vigil may see bursts of report-only traffic during policy rollouts.
- The `sandbox` directive is ignored in report-only mode and will never generate violation reports -- Vigil will not receive sandbox-related reports from report-only headers.
- Both `report-to` (modern, `application/reports+json`) and `report-uri` (legacy, `application/csp-report`) endpoints are relevant -- Vigil must handle both Content-Types as sites will use both directives for cross-browser compatibility.
