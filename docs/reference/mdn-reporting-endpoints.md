# Reporting-Endpoints Header -- MDN Reference

> Source: <https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Reporting-Endpoints>
> Spec Status: Living Standard
> Last Fetched: 2026-03-16

## Overview

The HTTP `Reporting-Endpoints` response header allows website administrators to specify one or more named endpoints where the browser should send reports generated by the Reporting API. These endpoints can receive CSP violation reports, Cross-Origin-Opener-Policy reports, and other generic violation reports. This header replaces the deprecated `Report-To` header and should be used in preference.

Header Type: Response header (not a CORS-safelisted response header).

## Syntax

```
Reporting-Endpoints: <endpoint>
Reporting-Endpoints: <endpoint>, ..., <endpointN>
```

Where each `<endpoint>` follows the format:

```
<endpoint-name>="<URL>"
```

## Parameters

| Parameter         | Description                                                                                                         |
| ----------------- | ------------------------------------------------------------------------------------------------------------------- |
| `<endpoint-name>` | An arbitrary name for the endpoint (e.g., `csp-endpoint`). This name is referenced by directives like CSP `report-to`. |
| `<URL>`           | A valid HTTPS URI enclosed in double quotes. This is the URL where the browser will POST reports. Non-secure (HTTP) endpoints are silently ignored by the browser. |

Multiple endpoints are separated by commas. Each endpoint defines a distinct name-to-URL mapping.

## Examples

### Single Endpoint

A CSP violation reporting configuration with a single endpoint:

```http
Reporting-Endpoints: csp-endpoint="https://example.com/csp-reports"
Content-Security-Policy: default-src 'self'; report-to csp-endpoint
```

### Multiple Endpoints

Different report types can be routed to different endpoints:

```http
Reporting-Endpoints: csp-endpoint="https://example.com/csp-reports",
                     permissions-endpoint="https://example.com/permissions-policy-reports"
```

## Security Requirements

- **HTTPS only**: Endpoint URLs must use HTTPS. Non-secure (plain HTTP) endpoints are silently ignored by the browser -- no reports will be sent to them.
- Endpoint URLs must be valid URIs enclosed in double-quoted strings.

## Replaces Deprecated Report-To Header

The `Reporting-Endpoints` header is the modern replacement for the deprecated `Report-To` header.

| Aspect           | `Report-To` (Deprecated, v0)                                | `Reporting-Endpoints` (Current, v1)             |
| ---------------- | ------------------------------------------------------------ | ----------------------------------------------- |
| Format           | JSON object with `group`, `max_age`, `endpoints` array      | Simple `name="url"` pairs                       |
| Endpoint Groups  | Supports endpoint groups with failover and priority          | Single URL per named endpoint                   |
| `max_age`        | Required (controls caching of endpoint config)               | Not applicable                                  |
| Browser Support  | Chrome 69+, Edge 69+                                        | Chrome 96+, Edge 96+                            |

During the transition period, sites may serve both headers simultaneously. Chrome/Edge 96+ will prefer `Reporting-Endpoints` over `Report-To` when both are present.

## Interaction with CSP report-to Directive

The `Reporting-Endpoints` header works in tandem with the CSP `report-to` directive:

1. The `Reporting-Endpoints` header defines one or more named endpoints with their URLs.
2. The `Content-Security-Policy` header's `report-to` directive references an endpoint by name.
3. When a CSP violation occurs, the browser resolves the endpoint name to its URL and sends the report there.

Both headers must be present in the **same HTTP response** for reporting to function. Without the corresponding `Reporting-Endpoints` header, the browser has no URL to send reports to and will silently discard them.

```http
Reporting-Endpoints: csp-endpoint="https://example.com/csp-reports"
Content-Security-Policy: default-src 'self'; report-to csp-endpoint
```

Reports are sent via HTTP POST with Content-Type `application/reports+json`. See the `mdn-csp-report-to.md` reference for the full report envelope and body field documentation.

## Browser Compatibility

MDN marks this feature as **Limited availability** -- it is not Baseline because it does not work in some of the most widely-used browsers.

| Browser           | Support           | Notes                                       |
| ----------------- | ----------------- | ------------------------------------------- |
| Chrome            | 96+               | Full support for Reporting API v1           |
| Edge              | 96+               | Full support (Chromium-based)               |
| Firefox           | Not supported     | Historically has not supported the Reporting API endpoints |
| Safari            | Not supported     | No objection to the spec, but no implementation yet |
| Opera             | 82+ (estimated)   | Chromium-based, follows Chrome              |

Because of limited adoption, sites should continue to specify both `report-uri` (legacy) and `report-to` (modern) in their CSP headers for full browser coverage.

Consult the [MDN compatibility table](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Reporting-Endpoints#browser_compatibility) for current data.

## Relevance to Vigil

- Vigil is the endpoint URL that site operators configure in their `Reporting-Endpoints` header (e.g., `Reporting-Endpoints: csp-endpoint="https://vigil.example.com/report"`).
- The endpoint URL must use HTTPS in production; browsers silently ignore HTTP endpoints.
- Multiple endpoint names can point to the same Vigil instance (e.g., `csp-endpoint="https://vigil.example.com/report", default="https://vigil.example.com/report"`).
- This header replaces the older `Report-To` header (JSON-based endpoint groups). Vigil should expect traffic from both the legacy and modern reporting formats during the transition period.
- Reports arriving via the modern `Reporting-Endpoints` path use `Content-Type: application/reports+json` with camelCase field names -- distinct from the legacy `application/csp-report` format with kebab-case names.
- Vigil already handles both formats (legacy `csp-report` + modern `report-to`) and stores raw JSON bodies, so no additional parsing is needed for this header's reports.
